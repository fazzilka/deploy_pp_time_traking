# Требования к системе учёта задач и интервалов времени

## 1. Назначение

Система предназначена для учёта задач и интервалов времени, затраченного на выполнение задач, с автоматическим вычислением суммарного времени. Система должна предоставлять пользовательский интерфейс и программный интерфейс для управления задачами, управления таймером и получения сводной информации по времени.

## 2. Термины и определения

- **Задача** — объект учёта работы, для которого измеряется затраченное время.
- **Интервал времени** — период выполнения задачи между запуском и остановкой измерения времени.
- **Активный интервал времени** — интервал времени без времени окончания.
- **Закрытый интервал времени** — интервал времени с указанными временем начала и временем окончания.
- **Суммарное время задачи** — сумма длительностей всех закрытых интервалов времени по задаче.
- **Сводка времени** — агрегированная информация о суммарном времени по всем задачам и перечне задач с наибольшим временем.

## 3. Функциональные требования

### 3.1. Управление задачами

#### 3.1.1. Создание задачи
Система должна предоставлять возможность создать задачу с указанием:
- названия;
- описания (необязательное).

Результат:
- задача получает уникальный идентификатор;
- суммарное время новой задачи равно нулю;
- список интервалов времени пуст.

#### 3.1.2. Получение списка задач
Система должна предоставлять возможность получить список задач.

Для списка задач система должна поддерживать:
- поиск по названию;
- фильтрацию по наличию суммарного времени больше нуля.

Результат:
- возвращается список задач с основными полями задачи и суммарным временем.

#### 3.1.3. Получение задачи по идентификатору
Система должна предоставлять возможность получить одну задачу по её идентификатору.

Результат:
- возвращаются все поля задачи, включая список интервалов времени и суммарное время.

#### 3.1.4. Обновление задачи
Система должна предоставлять возможность обновить задачу по её идентификатору.

Обновляемые поля:
- название;
- описание.

Результат:
- поля задачи изменены;
- интервалы времени и суммарное время не должны изменяться в результате обновления названия и описания.

#### 3.1.5. Удаление задачи
Система должна предоставлять возможность удалить задачу по её идентификатору.

Результат:
- задача удалена;
- все интервалы времени, связанные с задачей, удалены.

### 3.2. Управление таймером

#### 3.2.1. Запуск измерения времени
Система должна предоставлять возможность запустить измерение времени для выбранной задачи.

Правила:
- если у задачи нет активного интервала времени, создаётся новый активный интервал времени с заполненным временем начала;
- если у задачи уже есть активный интервал времени, запуск измерения должен быть отклонён.

Результат:
- создан активный интервал времени.

#### 3.2.2. Остановка измерения времени
Система должна предоставлять возможность остановить измерение времени для выбранной задачи.

Правила:
- если у задачи есть активный интервал времени, в него записывается время окончания, после чего интервал становится закрытым;
- если активного интервала времени нет, остановка измерения должна быть отклонена.

Результат:
- активный интервал времени закрыт;
- суммарное время задачи пересчитано.

### 3.3. Учёт интервалов времени

#### 3.3.1. Хранение интервалов времени
Система должна хранить список интервалов времени для каждой задачи.

Каждый интервал времени должен содержать:
- время начала;
- время окончания (может отсутствовать, если интервал активный).

#### 3.3.2. Вычисление суммарного времени задачи
Система должна вычислять суммарное время задачи как сумму длительностей всех закрытых интервалов времени.

Правила:
- длительность интервала времени равна разнице между временем окончания и временем начала;
- активный интервал времени не должен учитываться в суммарном времени до момента закрытия.

#### 3.3.3. Защита от некорректных интервалов времени
Система должна предотвращать создание некорректных интервалов времени:
- у задачи не должно быть более одного активного интервала времени;
- время окончания не должно быть меньше времени начала.

### 3.4. Поиск и фильтрация

#### 3.4.1. Поиск по названию
Система должна поддерживать поиск задач по названию по строке, введённой пользователем.

Результат:
- возвращаются задачи, название которых содержит указанную строку.

#### 3.4.2. Фильтрация по суммарному времени
Система должна поддерживать фильтр, который возвращает только задачи, у которых суммарное время больше нуля.

### 3.5. Сводка и статистика

#### 3.5.1. Получение сводки времени
Система должна предоставлять сводку времени, включающую:
- суммарное время по всем задачам;
- перечень задач с наибольшим временем.

#### 3.5.2. Перечень задач с наибольшим временем
Система должна возвращать перечень задач, отсортированный по убыванию суммарного времени, и ограничивать количество задач в перечне.

Значение ограничения по умолчанию — 10 задач.

## 4. Нефункциональные требования

### 4.1. Общие требования к качеству

1. Система должна обеспечивать корректность вычисления суммарного времени.
2. Система должна обеспечивать целостность данных при параллельных запросах запуска и остановки измерения времени.
3. Система должна быть устойчивой к повторным запросам, которые могут привести к дублированию активного интервала времени.

### 4.2. Производительность

1. Получение списка задач должно выполняться без заметных задержек при типичном учебном объёме данных.
2. Получение сводки времени должно выполняться с использованием агрегирования в базе данных.
3. Время ответа сервера на операции создания, чтения, обновления и удаления задач должно быть приемлемым для интерактивной работы пользователя.

### 4.3. Надёжность и целостность данных

1. Все изменения данных должны выполняться в транзакциях базы данных.
2. Должна быть обеспечена защита от состояния, при котором у одной задачи одновременно существует более одного активного интервала времени.
3. Должна быть обеспечена корректная обработка ошибок с понятными сообщениями.

### 4.4. Безопасность

1. Система не должна хранить персональные данные пользователя, кроме данных, необходимых для работы с задачами.
2. Внешние сетевые интерфейсы системы должны быть ограничены только необходимыми сервисами.
3. Доступ к служебным интерфейсам мониторинга должен быть возможен только в среде разработки или при наличии отдельной настройки.

### 4.5. Логирование и мониторинг

1. Серверная часть должна записывать логи в стандартный вывод.
2. Логи должны содержать:
   - время события;
   - уровень события;
   - идентификатор запроса;
   - название маршрута или операции;
   - сообщение об ошибке при наличии ошибки.
3. Система должна предоставлять метрики для Prometheus.
4. Должны быть обеспечены сбор и просмотр логов через Loki и Promtail.
5. Должны быть обеспечены просмотр и построение графиков метрик через Grafana.

### 4.6. Совместимость и развёртывание

1. Серверная часть должна быть реализована на Python версии 3.14 и выше.
2. Система должна поддерживать запуск в контейнерах с использованием Docker Compose.
3. Запуск, остановка и просмотр логов должны выполняться через цели Makefile.

### 4.7. Требования к документации

1. Документация должна быть оформлена в формате Markdown и размещена в каталоге `docs/`.
2. В документации запрещены сокращения и аббревиатуры.
3. Документация должна описывать:
   - назначение системы;
   - основные сущности и правила;
   - программный интерфейс приложения;
   - порядок запуска и проверки работоспособности.

## 5. Ограничения и допущения

1. Система рассчитана на учебный сценарий использования и не предполагает учёта прав доступа пользователей.
2. Система работает с временем в UTC и хранит суммарное время в секундах.
3. Система не требует использования брокеров сообщений и фоновых очередей для выполнения основных функций.
