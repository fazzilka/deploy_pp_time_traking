# Интерфейс программирования приложения

## 1. Назначение

Документ описывает программный интерфейс приложения для управления задачами, интервалами времени и получения сводной информации по времени.

## 2. Общие сведения

Программный интерфейс приложения предоставляет операции для:
- регистрации и входа пользователей;
- изменения пароля текущего пользователя;
- проверки JWT access token;
- создания, получения, изменения и удаления задач;
- запуска и остановки измерения времени по задаче;
- получения сводной информации по времени;
- управления рабочими пространствами и участниками команды;
- получения профиля пользователя и активности за год;
- выполнения административных операций;
- проверки работоспособности приложения;
- получения метрик для мониторинга.

## 3. Формат данных и правила обработки

### 3.1. Формат времени
- Время передаётся и возвращается в строковом виде в формате ISO 8601.
- Время хранится и обрабатывается в всемирном координированном времени.

### 3.2. Формат ошибок
При ошибках возвращается объект с полями:
- `error` — краткое описание ошибки;
- `details` — подробности, если они есть.

Пример:
```json
{
  "error": "Невозможно запустить таймер",
  "details": "У задачи уже есть активный интервал времени"
}
```

3.3. Коды результатов
401 — отсутствует или неверен токен, либо неверный email или пароль.
403 — пользователь неактивен или не имеет нужной роли.
200 — операция выполнена успешно.
201 — сущность создана.
204 — сущность удалена.
400 — неверные данные запроса.
404 — сущность не найдена.
409 — конфликт состояния, операция невозможна из-за правил предметной области.
500 — внутренняя ошибка приложения.

```json
  "full_name": "Дмитрий"

```

Ответ:


### 4.2. Вход

Запрос:

```http
POST /api/v1/auth/login
```

Тело запроса:

```json
{
  "email": "user@example.com",
  "password": "password123"
}
```

Ответ:

```json
{
  "access_token": "...",
  "token_type": "bearer",
  "user": {
    "id": 1,
    "email": "user@example.com",
    "username": "dmitry",
```

JWT передается в защищенные маршруты:

```http
Authorization: Bearer <access_token>
```

## 5. Операции управления задачами

Все маршруты задач требуют JWT. Пользователь получает задачи текущего workspace, если является активным участником workspace. При обращении к задаче вне доступных workspace возвращается `404`. Создавать задачи и запускать таймер могут `owner`, `team_lead` и `member`; `viewer` имеет только чтение.

5.1. Получение списка задач

Назначение: получить список задач с поддержкой поиска и фильтрации.

Параметры:

search — строка для поиска по названию.
has_time — фильтр по наличию суммарного времени больше нуля. Возможные значения: true, false.
priority — фильтр по приоритету. Возможные значения: lowest, low, medium, high, highest.
deadline_before — вернуть задачи со сроком выполнения не позже указанной даты.
deadline_after — вернуть задачи со сроком выполнения не раньше указанной даты.
project_id — вернуть задачи конкретного проекта.
without_project — вернуть задачи без проекта.
limit — ограничение количества задач. Значение по умолчанию: 50, максимум: 100.
offset — смещение выборки. Значение по умолчанию: 0.
workspace_id — идентификатор workspace. Если параметр не задан, используется персональное workspace пользователя или выбранный frontend workspace.

Запрос:

GET /api/v1/tasks?workspace_id=2&search=текст&has_time=true&priority=high&deadline_before=2026-06-01&limit=50&offset=0

Ответ:
```json
[
  {
    "id": 1,
    "title": "Сделать таймер",
    "description": "Реализовать запуск и остановку измерения времени",
    "total_time_seconds": 420,
    "deadline": "2026-05-30",
    "priority": "high",
    "workspace_id": 2,
    "created_by_id": 1,
    "assignee_id": 1,
    "project_id": 2,
    "project": {
      "id": 2,
      "name": "Разработка backend",
      "color": "#1f6feb"
    }
  }
]
```

5.2. Получение задачи по идентификатору

Назначение: получить полную информацию о задаче, включая интервалы времени.

Запрос:
```
GET /api/v1/tasks/{task_id}
```
Ответ:

```json
{
  "id": 1,
  "title": "Сделать таймер",
  "description": "Реализовать запуск и остановку измерения времени",
  "total_time_seconds": 420,
  "deadline": "2026-05-30",
  "priority": "high",
  "time_intervals": [
    {
      "id": 10,
      "started_at": "2026-03-11T08:05:00Z",
      "ended_at": "2026-03-11T08:12:00Z"
    }
  ]
}
```

5.3. Создание задачи

Назначение: создать новую задачу.

Запрос:
```
POST /api/v1/tasks
```
Тело запроса:
```json
{
  "title": "Новая задача",
    "description": "Описание задачи",
    "deadline": "2026-05-30",
    "priority": "high",
    "project_id": 2,
    "workspace_id": 2,
    "assignee_id": 1
}
```
Ответ:
```json
{
  "id": 5,
  "title": "Новая задача",
  "description": "Описание задачи",
  "total_time_seconds": 0,
  "deadline": "2026-05-30",
  "priority": "high",
  "workspace_id": 2,
  "time_intervals": []
}
```
Если `project_id` передан, проект должен принадлежать тому же `workspace_id`. Если `assignee_id` передан, пользователь должен быть активным участником workspace.

5.4. Изменение задачи

Назначение: изменить название, описание, срок выполнения и приоритет задачи.

Запрос:
```
PATCH /api/v1/tasks/{task_id}
```
Тело запроса:

```json
{
  "title": "Обновлённое название",
  "description": "Обновлённое описание",
  "deadline": "2026-06-01",
  "priority": "highest",
  "project_id": null,
  "assignee_id": 2
}
```
Ответ:
```json
{
  "id": 5,
  "title": "Обновлённое название",
  "description": "Обновлённое описание",
  "total_time_seconds": 0,
  "deadline": "2026-06-01",
  "priority": "highest"
}
```
`owner` и `team_lead` могут изменять задачи workspace. `member` может изменять задачи, где он указан как автор или исполнитель. `viewer` не может изменять задачи.

5.5. Удаление задачи

Назначение: удалить задачу и связанные интервалы времени.

Запрос:
```
DELETE /api/v1/tasks/{task_id}
```
Ответ:

код результата 204, тело отсутствует.

Удаление задач доступно `owner` и `team_lead`.

## 6. Операции таймера

6.1. Запуск измерения времени

Назначение: создать активный интервал времени для задачи.

Правило: у задачи не может быть более одного активного интервала времени. Запуск доступен активным участникам workspace с ролями `owner`, `team_lead` и `member`; в интервал времени записывается `user_id` пользователя, который запустил таймер.

Запрос:
```
POST /api/v1/tasks/{task_id}/timer/start
```
Ответ:
```json
{
  "task_id": 5,
  "active_interval": {
    "id": 22,
    "started_at": "2026-03-13T10:15:00Z",
    "ended_at": null
  }
}
```
Ошибка конфликта состояния:

код результата 409.

6.2. Остановка измерения времени

Назначение: закрыть активный интервал времени у задачи.

Правило: остановка возможна только при наличии активного интервала времени.

Запрос:
```
POST /api/v1/tasks/{task_id}/timer/stop
```
Ответ:
```json
{
  "task_id": 5,
  "closed_interval": {
    "id": 22,
    "started_at": "2026-03-13T10:15:00Z",
    "ended_at": "2026-03-13T10:20:00Z"
  },
  "total_time_seconds": 300
}
```
Ошибка конфликта состояния:

код результата 409.


## 8. Профиль и активность

### 8.1. Получение профиля

```http
GET /api/v1/users/me
```

Ответ содержит только базовые данные профиля без пересчёта статистики:

```json
{
  "id": 1,
  "email": "user@example.com",
  "username": "dmitry",
  "full_name": "Дмитрий",
  "role": "user",
  "is_active": true,
  "avatar_letter": "Д",
  "avatar_seed": "2fa6a6c4f8...",
  "created_at": "2026-05-18T10:00:00Z"
}
```

Backend возвращает `avatar_seed`, по которому frontend строит SVG-аватарку в стиле GitHub identicon. Поле `avatar_letter` сохраняется как fallback для старых клиентов и аварийного отображения. PNG/JPEG-файл аватара не хранится.

Для ответа используется приватное кэширование:

```http
Cache-Control: private, max-age=60
```

### 8.2. Обновление профиля

```http
PATCH /api/v1/users/me
```

Разрешены поля `username` и `full_name`.

### 8.3. Регенерация аватарки

```http
POST /api/v1/users/me/avatar/regenerate
```

Endpoint генерирует новый `avatar_seed` для текущего пользователя и возвращает обновлённый лёгкий профиль:

```json
{
  "id": 1,
  "email": "user@example.com",
  "username": "dmitry",
  "full_name": "Дмитрий",
  "role": "user",
  "is_active": true,
  "avatar_letter": "Д",
  "avatar_seed": "new-stable-seed",
  "created_at": "2026-05-18T10:00:00Z"
}
```

### 8.4. Получение статистики профиля

```http
GET /api/v1/users/me/stats
```

Ответ:

```json
{
  "tasks_count": 12,
  "tasks_with_time_count": 7,
  "total_time_seconds": 18420,
  "current_streak_days": 4,
  "max_streak_days": 16
}
```

Статистика считается отдельно от базового профиля через SQL-агрегации и activity-агрегацию по дням. Для ответа используется:

```http
Cache-Control: private, max-age=15
```

### 8.5. Изменение пароля

```http
POST /api/v1/users/me/change-password
```

JWT обязателен. Тело запроса:

```json
{
  "old_password": "old-password",
  "new_password": "new-password",
  "confirm_password": "new-password"
}
```

Правила:

- старый пароль обязателен и проверяется на backend;
- новый пароль должен быть не короче 6 символов;
- новый пароль и подтверждение должны совпадать;
- новый пароль должен отличаться от старого;
- пароль хранится только в виде hash;
- текущая JWT-сессия после успешной смены пароля остаётся активной.

Ответ:

```json
{
  "message": "Пароль успешно изменён"
}
```

Ошибки:

- `400`, если старый пароль указан неверно или новый пароль совпадает со старым;
- `422`, если новый пароль короткий или подтверждение не совпадает.

### 8.6. Активность за год

```http
GET /api/v1/users/me/activity?year=2026
```

Ответ содержит все дни года или последних 365 дней, количество закрытых интервалов, суммарное время и `level` от 0 до 4.

Для ответа используется:

```http
Cache-Control: private, max-age=30
```

## 9. Проекты

Маршруты проектов требуют JWT. Пользователь видит проекты workspace, где он является активным участником. Создание, изменение и архивирование проектов доступны только ролям `owner` и `team_lead`.

### 9.1. Получение списка проектов

```http
GET /api/v1/projects?workspace_id=2&include_archived=false&search=backend
```

Ответ:

```json
[
  {
    "id": 2,
    "name": "Разработка backend",
    "description": "API и база данных",
    "color": "#1f6feb",
    "is_archived": false,
    "created_at": "2026-06-01T10:00:00Z",
    "updated_at": "2026-06-01T10:00:00Z",
    "tasks_count": 4,
    "active_tasks_count": 1,
    "tasks_with_time_count": 3,
    "total_time_seconds": 18420
  }
]
```

Архивные проекты скрываются по умолчанию.

### 9.2. Создание проекта

```http
POST /api/v1/projects
```

```json
{
  "name": "Учёба",
  "description": "Курсы и домашние задания",
  "color": "#2ea043",
  "workspace_id": 2
}
```

Название проекта уникально в пределах пользователя. При повторе возвращается `409`.
Для командной работы проект создаётся внутри workspace через поле `workspace_id`. Если поле не передано, используется текущий/default workspace.

### 9.3. Получение, изменение и архивирование проекта

```http
GET /api/v1/projects/{project_id}
PATCH /api/v1/projects/{project_id}
DELETE /api/v1/projects/{project_id}
```

`DELETE` архивирует проект и возвращает `204`. Задачи и история времени сохраняются.

### 9.4. Задачи проекта

```http
GET /api/v1/projects/{project_id}/tasks
```

Поддерживаются те же фильтры, что и у `GET /api/v1/tasks`: `search`, `has_time`, `priority`, `deadline_before`, `deadline_after`, `limit`, `offset`.

### 9.5. Сводка проекта

```http
GET /api/v1/projects/{project_id}/summary?limit=5
```

Ответ содержит количество задач, активные задачи, задачи с временем, суммарное время и `top_tasks`.

### 9.6. Время по проектам

```http
GET /api/v1/summary/projects
```

Ответ:

```json
{
  "items": [
    {
      "project_id": 2,
      "name": "Разработка backend",
      "color": "#1f6feb",
      "tasks_count": 4,
      "active_tasks_count": 1,
      "total_time_seconds": 18420,
      "percentage": 74.5
    },
    {
      "project_id": null,
      "name": "Без проекта",
      "color": "#8b949e",
      "tasks_count": 1,
      "active_tasks_count": 0,
      "total_time_seconds": 6300,
      "percentage": 25.5
    }
  ],
  "total_time_seconds": 24720
}
```

Группа “Без проекта” виртуальная и формируется для задач с `project_id = null`.

## 10. Рабочие пространства и команда

Все маршруты `/api/v1/workspaces` требуют JWT. Пользователь видит только workspace, где он является активным участником.

### 10.1. Список workspace

```http
GET /api/v1/workspaces
```

Ответ:

```json
[
  {
    "id": 1,
    "name": "Личное пространство",
    "description": null,
    "type": "personal",
    "owner_id": 1,
    "members_count": 1,
    "projects_count": 3,
    "tasks_count": 12,
    "total_time_seconds": 3600,
    "current_user_role": "owner"
  }
]
```

### 10.2. Создание team workspace

```http
POST /api/v1/workspaces
```

```json
{
  "name": "Engineering",
  "description": "Командное пространство разработки"
}
```

Создатель становится `owner`, а в `workspace_members` создаётся активная запись владельца. Персональное workspace создаётся автоматически при регистрации пользователя.

### 10.3. Получение и изменение workspace

```http
GET /api/v1/workspaces/{workspace_id}
PATCH /api/v1/workspaces/{workspace_id}
```

Изменение настроек доступно владельцу workspace. Удаление workspace в текущей версии отключено и возвращает `405`.

### 10.4. Участники workspace

```http
GET /api/v1/workspaces/{workspace_id}/members
POST /api/v1/workspaces/{workspace_id}/members
PATCH /api/v1/workspaces/{workspace_id}/members/{member_id}
DELETE /api/v1/workspaces/{workspace_id}/members/{member_id}
```

Добавление участника выполняется по email уже зарегистрированного пользователя и создаёт запись `workspace_members`:

```json
{
  "email": "member@example.com",
  "role": "member"
}
```

Если пользователь с email не найден, возвращается `404`. Если пользователь уже состоит в workspace, возвращается `409`. `Owner` может добавить `team_lead`, `member` или `viewer`; `team_lead` может добавить только `member` или `viewer`; назначение `owner` через этот endpoint запрещено.

### 10.5. Сводка workspace и участников

```http
GET /api/v1/workspaces/{workspace_id}/summary
GET /api/v1/workspaces/{workspace_id}/members/summary
```

Сводка workspace возвращает количество участников, активных участников, проектов, активных проектов, задач, активных задач, завершённых задач и суммарное время. Сводка участников возвращает роль, статус, количество проектов, задач, завершённых задач и время по каждому пользователю. Время участника считается по `time_intervals.user_id`.

Email invite links, pending invitations, accept invite page и email-отправка в текущей версии не реализованы.

## 11. Административные маршруты

Все маршруты `/api/v1/admin/...` доступны только пользователю с ролью `admin`.

Основные маршруты:

- `GET /api/v1/admin/users`;
- `GET /api/v1/admin/users/{user_id}`;
- `PATCH /api/v1/admin/users/{user_id}`;
- `GET /api/v1/admin/users/{user_id}/activity`;
- `GET /api/v1/admin/stats`.

## 12. Служебные операции

12.1. Проверка работоспособности

Назначение: проверить работоспособность приложения и доступность базы данных.

Запрос:
```
GET /health
```
Ответ:
```json
{
  "status": "ok"
}
```
12.2. Получение метрик

Назначение: получить метрики приложения для мониторинга.

Запрос:
```
GET /metrics
```
Ответ:

текстовый формат метрик.
