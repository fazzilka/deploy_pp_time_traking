# Руководство по запуску

## 1. Назначение

Руководство описывает порядок запуска серверной части приложения, а также способы проверки работоспособности, просмотра логов и метрик мониторинга.

## 2. Предварительные требования

Для запуска должны быть установлены:
- система контроля версий Git;
- среда контейнеризации Docker;
- инструмент управления контейнерами Docker Compose;
- утилита make.

## 3. Структура проекта

В корне проекта находятся каталоги:
- `server` — серверная часть приложения;
- `Frontend` — клиентская часть приложения;
- `docs` — документация;
- `samples` — примеры данных и запросов.

## 4. Запуск приложения

1. Перейти в каталог `server`:
```bash
cd server
```
Запустить приложение:
```bash
make run
```
Убедиться в запуске:

- выполнить запрос проверки работоспособности:
```
GET /health
```
5. Остановка приложения

Для остановки выполнить:
```bash
cd server
make down
```
6. Просмотр логов

Для просмотра логов серверной части выполнить:
```bash
cd server
make logs
```
7. Проверка программного интерфейса приложения

Примеры запросов находятся в файле:

- samples/example_requests.http

Запросы можно выполнять любым клиентом, который поддерживает отправку запросов к программному интерфейсу приложения.

8. Мониторинг и просмотр логов

После запуска доступны компоненты наблюдаемости:

 1. Метрики приложения доступны по маршруту:
```
GET /metrics
```
 2. Визуализация метрик и просмотр логов доступны через Grafana.
  - Адрес Grafana указывается в настройках контейнеров и зависит от выбранных портов.
 3. Сбор метрик выполняется Prometheus.
  - Источником метрик является маршрут /metrics серверной части.
 4. Сбор и хранение логов выполняется связкой Promtail и Loki.
  - Promtail читает логи контейнеров и отправляет их в Loki.
  - Loki хранит логи и предоставляет доступ для Grafana.

9. Уведомления, RabbitMQ и Celery

Система уведомлений использует:

- PostgreSQL — хранение уведомлений, статуса прочтения, dedupe key и delivery status;
- RabbitMQ — только broker фоновых задач Celery;
- `celery-worker` — создание deadline-уведомлений и delivery через email/Telegram;
- `celery-beat` — периодический запуск `scan_deadline_notifications` раз в 60 секунд.

Локальный compose backend находится в `server/Backend`:

```bash
cd server/Backend
docker compose up -d --build
```

RabbitMQ management UI в локальном compose доступен на:

```text
http://127.0.0.1:15672
```

Email и Telegram выключены по умолчанию. In-app уведомления работают сразу после миграций, потому что frontend читает их из backend API.

10. Типовые команды обслуживания
- Проверка форматирования и статического анализа:
```bash
cd server
make lint
```

Проверка типов:
```bash
cd server
make type
```
Установка проверок перед коммитом:
```bash
cd server
make hooks
```
Применение миграций базы данных:
```bash
cd server
make upgrade
```

Основные переменные окружения уведомлений:

```env
CELERY_BROKER_URL=amqp://time_tracking:time_tracking@rabbitmq:5672//
CELERY_TASK_ALWAYS_EAGER=false
APP_TIMEZONE=Europe/Moscow
DEADLINE_REMINDER_MINUTES=60
OVERDUE_NOTIFICATION_LOOKBACK_HOURS=24
EMAIL_NOTIFICATIONS_ENABLED=false
SMTP_HOST=
SMTP_PORT=587
SMTP_USERNAME=
SMTP_PASSWORD=
SMTP_FROM_EMAIL=
SMTP_FROM_NAME=Time Tracking
SMTP_USE_TLS=true
SMTP_USE_SSL=false
TELEGRAM_NOTIFICATIONS_ENABLED=false
TELEGRAM_BOT_TOKEN=
APP_PUBLIC_URL=https://time-tracking.online
```

Deadline-уведомления сравниваются в UTC. Frontend отправляет deadline как ISO datetime с timezone, backend хранит UTC instant. `APP_TIMEZONE` используется только для совместимости со старыми date-only значениями и для читаемого текста уведомлений. `deadline_soon` создаётся только до дедлайна, а `deadline_overdue` создаётся после дедлайна один раз на конкретный deadline instant. `OVERDUE_NOTIFICATION_LOOKBACK_HOURS` ограничивает автоматическое создание overdue-уведомлений для старых задач после деплоя.
