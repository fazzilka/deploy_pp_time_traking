# Авторизация, роли, профиль и активность

## 1. Назначение

Серверная часть поддерживает регистрацию пользователей, вход по email и паролю, JWT access token, роли `user` и `admin`, профиль текущего пользователя и календарную активность за год.

Frontend не передает `user_id` при создании задач. Backend определяет владельца задачи по JWT и отдает пользователю только его данные.

## 2. Регистрация

Маршрут:

```http
POST /api/v1/auth/register
```

Тело запроса:

```json
{
  "email": "user@example.com",
  "username": "dmitry",
  "password": "password123",
  "full_name": "Дмитрий"
}
```

Правила:

- `email` уникален.
- `username` уникален.
- пароль должен быть не короче 6 символов;
- пароль хранится только в виде bcrypt-хеша;
- frontend запрашивает подтверждение пароля при регистрации и не отправляет запрос, если пароли не совпадают;
- роль при обычной регистрации всегда `user`;
- поле `role` в запросе регистрации запрещено.

## 3. Вход и JWT

Маршрут:

```http
POST /api/v1/auth/login
```

Тело запроса:

```json
{
  "email": "user@example.com",
  "password": "password123"
}
```

Ответ содержит JWT access token:

```json
{
  "access_token": "...",
  "token_type": "bearer",
  "user": {
    "id": 1,
    "email": "user@example.com",
    "username": "dmitry",
    "full_name": "Дмитрий",
    "role": "user",
    "is_active": true,
    "avatar_letter": "Д"
  }
}
```

JWT содержит:

- `sub` — идентификатор пользователя;
- `role` — роль пользователя;
- `exp` — срок действия токена.

Настройки:

- `JWT_SECRET_KEY`;
- `JWT_ALGORITHM`;
- `ACCESS_TOKEN_EXPIRE_MINUTES`.

## 4. Роли

Роль `user`:

- управляет только своими задачами;
- запускает и останавливает таймер только по своим задачам;
- видит только свою сводку;
- видит свой профиль и активность.

Роль `admin`:

- имеет доступ к `/api/v1/admin/...`;
- получает список пользователей;
- получает профиль пользователя по идентификатору;
- получает активность пользователя;
- получает общую статистику системы;
- меняет роль, активность, username и full_name пользователя.

Обычный пользователь не имеет доступа к admin-маршрутам.

## 5. Защищенные маршруты

JWT обязателен для:

- `GET /api/v1/tasks`;
- `GET /api/v1/tasks/{task_id}`;
- `POST /api/v1/tasks`;
- `PATCH /api/v1/tasks/{task_id}`;
- `DELETE /api/v1/tasks/{task_id}`;
- `POST /api/v1/tasks/{task_id}/timer/start`;
- `POST /api/v1/tasks/{task_id}/timer/stop`;
- `GET /api/v1/summary`;
- `GET /api/v1/users/me`;
- `PATCH /api/v1/users/me`;
- `POST /api/v1/users/me/change-password`;
- `GET /api/v1/users/me/activity`;
- все маршруты `/api/v1/admin/...`.

Если пользователь обращается к чужой задаче, backend возвращает `404`, чтобы не раскрывать существование чужой записи.

## 6. Профиль

Маршрут:

```http
GET /api/v1/users/me
```

Ответ:

```json
{
  "id": 1,
  "email": "user@example.com",
  "username": "dmitry",
  "full_name": "Дмитрий",
  "role": "user",
  "is_active": true,
  "avatar_letter": "Д",
  "created_at": "2026-05-18T10:00:00Z",
  "stats": {
    "tasks_count": 12,
    "tasks_with_time_count": 7,
    "total_time_seconds": 18420,
    "current_streak_days": 4,
    "max_streak_days": 16
  }
}
```

`avatar_letter` вычисляется на backend:

- если есть `full_name`, берется первая буква `full_name`;
- иначе берется первая буква `username`;
- буква приводится к верхнему регистру.

Backend не хранит и не возвращает `avatar_color`. Цвет аватара выбирает frontend.

Обновление профиля:

```http
PATCH /api/v1/users/me
```

Разрешены только поля:

- `username`;
- `full_name`.

Email, пароль, роль, аватар и цвет аватара этим маршрутом не меняются.

## 7. Изменение пароля

Маршрут:

```http
POST /api/v1/users/me/change-password
```

Тело запроса:

```json
{
  "old_password": "old-password",
  "new_password": "new-password",
  "confirm_password": "new-password"
}
```

Правила:

- маршрут требует JWT активного пользователя;
- старый пароль проверяется через существующий hash;
- новый пароль должен быть не короче 6 символов;
- новый пароль и подтверждение должны совпадать;
- новый пароль должен отличаться от старого;
- backend сохраняет только hash нового пароля;
- текущий JWT после смены пароля остаётся активным.

Успешный ответ:

```json
{
  "message": "Пароль успешно изменён"
}
```

Ошибки:

- `400`, если старый пароль неверный или новый пароль совпадает со старым;
- `422`, если новый пароль короткий или подтверждение не совпадает.

## 8. Активность за год

Маршрут текущего пользователя:

```http
GET /api/v1/users/me/activity?year=2026
```

Маршрут администратора:

```http
GET /api/v1/admin/users/{user_id}/activity?year=2026
```

Если `year` не передан, возвращаются последние 365 дней.

День считается активным, если в этот день у пользователя есть хотя бы один закрытый интервал времени. Незавершенные интервалы, где `finished_at = null`, не учитываются.

Ответ содержит все дни периода, включая дни без активности:

```json
{
  "year": 2026,
  "days": [
    {
      "date": "2026-01-01",
      "intervals_count": 0,
      "total_time_seconds": 0,
      "level": 0
    }
  ],
  "summary": {
    "active_days_count": 43,
    "current_streak_days": 4,
    "max_streak_days": 16,
    "total_intervals_count": 120,
    "total_time_seconds": 98400
  }
}
```

Уровни активности:

- `0` — 0 секунд;
- `1` — от 1 до 1799 секунд;
- `2` — от 1800 до 3599 секунд;
- `3` — от 3600 до 7199 секунд;
- `4` — 7200 секунд и больше.

## 9. Административные маршруты

Префикс:

```http
/api/v1/admin
```

Маршруты:

- `GET /api/v1/admin/users`;
- `GET /api/v1/admin/users/{user_id}`;
- `PATCH /api/v1/admin/users/{user_id}`;
- `GET /api/v1/admin/users/{user_id}/activity`;
- `GET /api/v1/admin/stats`.

Административные ответы не возвращают `hashed_password`.

## 10. Создание первого администратора

Администратор не создается автоматически.

Команда:

```bash
make create-admin
```

Переменные окружения:

- `ADMIN_EMAIL`;
- `ADMIN_USERNAME`;
- `ADMIN_PASSWORD`;
- `ADMIN_FULL_NAME`.

Скрипт проверяет дубликаты по email и username. Если пользователь уже существует, повторная запись не создается.
