# Оптимизация производительности

## 1. Цель

Цель изменений — снизить нагрузку на базу данных и уменьшить количество повторных запросов между frontend и backend без изменения существующих базовых контрактов приложения.

## 2. Изученные части проекта

- `README.md`, `server/Backend/README.md`, `server/frontend/README.md`;
- документы `docs/01`-`docs/15`, включая модель данных, требования, авторизацию, программный интерфейс, отчёт о связке frontend и backend;
- `samples/example_requests.http` и `samples/example_data.json`;
- FastAPI routers, SQLAlchemy models, Pydantic schemas, services, Alembic migrations, database session, auth dependencies, CORS, Makefile и `pyproject.toml`;
- frontend API layer, страницы Dashboard, Profile, Reports, Navigation и компоненты задач, таймера и activity grid.

## 3. Найденные проблемы

- список задач поддерживал пагинацию, но значения по умолчанию были слишком большими для интерактивного экрана;
- detail endpoint задачи загружал только активные интервалы, хотя детальный контракт допускает полную историю;
- profile stats выполнял несколько отдельных агрегирующих запросов по задачам;
- Reports и Profile получали полный список задач там, где достаточно компактной сводки;
- после start, stop и delete Dashboard повторно запрашивал список задач;
- не хватало составных индексов под фильтрацию задач по пользователю и под activity range по интервалам.

## 4. Выполненные изменения

- `GET /api/v1/tasks` оставлен в формате массива, добавлены значения `limit=50`, максимум `100`, `offset=0`;
- list endpoint задач загружает только активные интервалы, нужные для восстановления запущенных таймеров;
- detail endpoint задачи возвращает полную историю интервалов;
- stop timer сохраняет инкрементальное обновление `total_time_seconds`;
- summary возвращает компактные top tasks с описанием, сроком, приоритетом и `tasks_with_time_count`;
- Profile и Reports используют `GET /api/v1/summary?limit=3` вместо полного списка задач;
- Dashboard обновляет локальное состояние после start, stop и delete без повторного `GET /tasks`;
- одинаковые одновременные GET-запросы frontend объединяются на уровне in-flight request.

## 5. Оптимизация запросов к базе данных

- Summary считает общую сумму и количество задач с временем агрегатами SQL.
- Top tasks ограничиваются через SQL `LIMIT`.
- Profile stats считает `tasks_count`, `tasks_with_time_count` и `total_time_seconds` одним SQL-запросом.
- Activity grid использует один агрегирующий запрос по дням и заполняет пустые дни в Python.
- Список задач использует `selectinload` только для активных интервалов.

## 6. Добавленные индексы

Для `tasks`:
- `created_at`, `updated_at`;
- `user_id + created_at`;
- `user_id + deadline`;
- `user_id + priority`;
- `user_id + total_time_seconds`;
- `workspace_id + is_completed`;
- `workspace_id + deadline`.

Для `time_intervals`:
- `task_id`;
- `started_at`;
- `finished_at`;
- `task_id + started_at`;
- `started_at + finished_at`.

Для `users`:
- `is_active`.

Для `notifications`:
- `user_id + is_read + created_at`.

## 7. Изменения во frontend

- Dashboard отправляет `limit=50&offset=0` при загрузке задач.
- Поиск по задачам остаётся с debounce 300 мс.
- Live timer продолжает обновляться локально через `setInterval`, без backend polling.
- Start, stop и delete обновляют локальный список задач из результата операции.
- Profile и Reports больше не загружают полный task list для top tasks и количества задач с временем.
- Для одинаковых параллельных `GET /tasks`, `GET /summary`, `GET /users/me` и `GET /users/me/activity` используется in-flight deduplication.

### Workspace и team updates

- `WorkspaceContext` является единым источником списка workspace и текущего workspace.
- Создание и изменение workspace обновляет локальный state без полного повторного запроса списка.
- Workspace list обновляется по `workspace.membership.changed` через `GET /api/v1/events/stream`.
- Для Bearer JWT используется `fetch` streaming, а не `EventSource`, чтобы не передавать токен в URL.
- Если stream оборвался, frontend переподключается с backoff до 30 секунд.
- Fallback polling workspace list выполняется раз в 60 секунд и не включает глобальный loading state.
- TeamPage после add/remove/update role обновляет локальный список участников и счётчики; полный reload используется только как targeted recovery после внешнего события или ошибки.

### Backend workspace list

- `GET /api/v1/workspaces` больше не строит каждый workspace через отдельные запросы `members_count`, `projects_count`, `tasks_count` и `total_time_seconds`.
- Backend получает доступные workspace пользователя одним запросом, а счётчики считает grouped SQL aggregates по всем workspace сразу.
- Это снижает количество SQL-запросов для пользователя с несколькими организациями с `1 + N * 4` до фиксированного набора grouped queries.

### Оптимизация запросов и кэширование

- `GET /api/v1/users/me` возвращает только базовые данные профиля без `stats`.
- `GET /api/v1/users/me/stats` возвращает динамическую статистику отдельным endpoint.
- Frontend больше не обновляет profile, stats, summary и activity по TTL; кэш обновляется только по событиям изменения данных.
- Для profile, stats, summary и activity используется состояние `loaded/dirty/pending`: повторный переход на страницу возвращает кэш, а параллельные запросы склеиваются в один Promise.
- После login кэш базового профиля заполняется из ответа авторизации; stats cache заполняется только если `stats` реально пришли в ответе.
- После logout кэш профиля, stats, activity и reports очищается.
- После создания задачи инвалидируются stats и reports.
- После удаления задачи или остановки таймера инвалидируются stats, activity и reports.
- После обновления задачи инвалидируется reports cache.
- Activity grid не запрашивается вместе с каждым `GET /users/me` и обновляется отдельно.
- Backend добавляет `Cache-Control: private` для profile, stats и activity.
- Для JSON-ответов больше 1000 байт включено gzip-сжатие.
- `POST /api/v1/auth/login` возвращает lightweight user без `stats`, поэтому вход не запускает profile/activity агрегации.
- Проверка пароля и хеширование при регистрации выполняются в threadpool, чтобы bcrypt не блокировал event loop.
- Проверка уникальности email/username при регистрации выполняется одним indexed query.
- Для auth добавлены Prometheus histogram metrics `auth_login_duration_seconds` и `auth_register_duration_seconds` со стадиями `db_lookup`, `password_verify`, `password_hash`, `db_insert`, `token_create`, `total`.

## 8. Проверка backend

Проверяются:
- пагинация списка задач;
- фильтры поиска, времени, приоритета и срока;
- summary с ограничением top tasks;
- отсутствие доступа к чужим задачам через `404`;
- один агрегирующий запрос activity;
- уменьшение числа агрегирующих запросов profile stats;
- инкрементальное обновление времени при stop timer.

## 9. Проверка frontend

Проверяются:
- TypeScript typecheck;
- production build;
- отсутствие повторного получения списка задач после start, stop и delete;
- отсутствие запросов каждую секунду для live timer;
- получение Profile и Reports через summary/activity без полного task list.

## 10. Итог

Основная нагрузка снижена за счёт SQL-агрегаций, индексов, меньших response для списка задач и удаления повторных frontend-запросов. Контракт `GET /api/v1/tasks` сохранён как массив, а новые поля summary добавлены расширением ответа без удаления существующих данных.
