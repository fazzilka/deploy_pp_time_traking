# Мониторинг и наблюдаемость

## 1. Назначение

Документ описывает production-наблюдаемость проекта Time Tracking: сбор метрик, сбор логов, источники данных Grafana и dashboard для диагностики состояния приложения.

## 2. Компоненты

В production stack используются:

- Prometheus — сбор и хранение метрик;
- Grafana — визуализация метрик и логов;
- Loki — хранение логов;
- Promtail — доставка Docker logs в Loki;
- postgres-exporter — метрики PostgreSQL;
- cAdvisor — метрики контейнеров;
- FastAPI Instrumentator — HTTP-метрики backend;
- SQLAlchemy events — метрики выполнения SQL-запросов backend;
- RabbitMQ — broker Celery задач уведомлений;
- Celery worker и Celery Beat — фоновые задачи уведомлений.

## 3. Datasource Grafana

Grafana provisioning добавляет два datasource:

- `Prometheus` — `http://prometheus:9090`;
- `Loki` — `http://loki:3100`.

Dashboard находится в:

```text
deploy/grafana/dashboards/time-tracking-observability.json
```

Название dashboard:

```text
Time Tracking Observability
```

## 4. HTTP-метрики backend

Backend отдаёт метрики на маршруте:

```http
GET /metrics
```

Основные HTTP-метрики:

- `http_requests_total{method,status,handler}` — количество запросов по методу, статусу и handler;
- `http_request_duration_highr_seconds` — общая latency с более точными bucket для p50/p95/p99;
- `http_request_duration_seconds{method,handler}` — latency по endpoint;
- `http_request_size_bytes{handler}` — размер входящих запросов;
- `http_response_size_bytes{handler}` — размер ответов.

Статусы сгруппированы как `2xx`, `4xx`, `5xx`.

Для endpoint latency в backend настроены более плотные histogram buckets в диапазоне
`0.1–1.0s`. Это важно для панелей p95 по отдельным маршрутам: при слишком грубых
buckets, например `0.5s -> 1.0s`, `histogram_quantile` может показывать значение около
`0.95s` даже тогда, когда реальные запросы находятся немного выше `0.5s`.

## 5. Метрики SQL-запросов

Backend добавляет безопасные метрики SQLAlchemy без полного SQL-текста и без пользовательских identifiers:

- `db_query_duration_seconds{operation,database}` — histogram времени SQL-запросов;
- `db_queries_total{operation,database}` — количество SQL-запросов;
- `db_query_errors_total{operation,database}` — количество ошибок SQL-запросов.

Допустимые значения `operation`:

- `SELECT`;
- `INSERT`;
- `UPDATE`;
- `DELETE`;
- `OTHER`.

Эти метрики помогают понять, находится ли деградация в API-слое или в работе базы данных.

## 6. Метрики PostgreSQL

`postgres-exporter` собирает метрики PostgreSQL внутри Docker network.

Основные панели dashboard используют:

- `pg_up` — доступность PostgreSQL exporter;
- `pg_stat_activity_count` — количество подключений;
- `pg_database_size_bytes` — размер базы;
- `pg_stat_database_xact_commit` — commits;
- `pg_stat_database_xact_rollback` — rollbacks;
- `pg_stat_database_deadlocks` — deadlocks;
- `pg_stat_database_blks_hit` и `pg_stat_database_blks_read` — cache hit ratio;
- `pg_stat_database_tup_inserted`, `pg_stat_database_tup_updated`, `pg_stat_database_tup_deleted` — изменение строк.

## 7. Метрики контейнеров

cAdvisor собирает инфраструктурные метрики контейнеров:

- `container_cpu_usage_seconds_total` — CPU usage;
- `container_memory_usage_bytes` — memory usage;
- `container_network_receive_bytes_total` — входящий трафик;
- `container_network_transmit_bytes_total` — исходящий трафик.

Порты postgres-exporter и cAdvisor наружу не публикуются, Prometheus обращается к ним внутри Docker network.

## 8. Логи

Promtail читает Docker logs контейнеров с label:

```yaml
logging: promtail
```

Логи backend доступны в Loki:

```logql
{service="backend"}
```

Backend пишет JSON-логи с полями:

- `timestamp`;
- `level`;
- `logger`;
- `message`;
- `request_id`;
- `exception`, если есть исключение.

Ошибки backend:

```logql
{service="backend"} | json | level="ERROR"
```

Логи frontend/nginx:

```logql
{service="frontend"}
```

Логи фоновых уведомлений:

```logql
{service="celery-worker"}
{service="celery-beat"}
```

Worker пишет события scan/delivery без SMTP password и Telegram token. Повторный запуск scan-задачи безопасен: dedupe key в PostgreSQL защищает от дублей.

Для deadline scan в Loki полезны события:

```logql
{service="celery-worker"} |= "deadline_scan_started"
{service="celery-worker"} |= "deadline_scan_finished"
{service="celery-worker"} |= "deadline_soon_created"
{service="celery-worker"} |= "deadline_overdue_created"
{service="celery-worker"} |= "deadline_notification_skipped_duplicate"
```

В structured logs указываются `task_id`, `user_id`, UTC deadline, UTC now, `remaining_seconds`, `overdue_seconds`, `reminder_minutes` и `overdue_lookback_hours`.

Логи пользовательских событий:

```logql
{service="backend"} |= "sse_client_connected"
{service="backend"} |= "sse_client_disconnected"
{service="backend"} |= "sse_event_sent"
{service="backend"} |= "sse_event_failed"
```

События stream не содержат JWT, SMTP password, Telegram token и не логируют полный URL запроса. При нескольких backend replicas текущий in-memory fanout нужно заменить на общий pub/sub слой.

Ошибки проксирования API:

```logql
{service="frontend"} |= "upstream"
```

## 9. Как открыть Grafana

Grafana в production compose привязана к `127.0.0.1:3000`.

Открыть через SSH tunnel:

```bash
ssh -L 3000:127.0.0.1:3000 root@SERVER_IP_OR_DOMAIN
```

После этого открыть:

```text
http://localhost:3000
```

Логин:

```text
admin
```

Пароль берётся из `.env`:

```env
GRAFANA_ADMIN_PASSWORD=...
```

## 10. Что смотреть при проблемах

### Backend down

Смотреть:

- `Backend up`;
- Prometheus target `backend`;
- backend logs;
- healthcheck backend container.

### Уведомления не создаются или не доставляются

Смотреть:

- статус контейнеров `rabbitmq`, `celery-worker`, `celery-beat`;
- логи `{service="celery-beat"}` для запуска периодического scan;
- логи `{service="celery-worker"}` для ошибок deadline scan и delivery;
- таблицы `notifications` и `notification_deliveries` в PostgreSQL;
- настройки `EMAIL_NOTIFICATIONS_ENABLED`, `SMTP_*`, `TELEGRAM_NOTIFICATIONS_ENABLED`, `TELEGRAM_BOT_TOKEN`.

Если email/Telegram выключены, delivery получает статус `skipped` или задача доставки не ставится; это нормальный режим.

### Высокая latency API

Смотреть:

- `HTTP p95 latency`;
- `p95 latency by endpoint`;
- `Slowest endpoints p95`;
- `DB query p95`;
- `DB query average latency`.

Если HTTP latency растёт вместе с DB latency, проблема вероятно в базе или SQL-запросах.

### Рост 5xx

Смотреть:

- `5xx rate`;
- `5xx errors by endpoint`;
- `Failed requests table`;
- `Backend errors`;
- `Recent exceptions`.

### Много 4xx

Смотреть:

- `4xx errors by endpoint`;
- `Status code distribution`;
- frontend/nginx logs.

Это может быть нормальным при неверных JWT, валидационных ошибках или внешних сканерах.

### Проблемы PostgreSQL

Смотреть:

- `PostgreSQL up`;
- `Active connections`;
- `Rollbacks / sec`;
- `Deadlocks`;
- `Cache hit ratio`;
- `Rows inserted / updated / deleted`.

### Высокий CPU или memory

Смотреть:

- `CPU usage by container`;
- `Memory usage by container`;
- `Container CPU top`;
- `Container memory top`.

Если backend CPU высокий вместе с большим request rate, проблема может быть в нагрузке на API. Если PostgreSQL memory/CPU растут вместе с DB latency, нужно проверять запросы и индексы.

## 11. Проверка после деплоя

Запуск production stack:

```bash
docker compose -f docker-compose.prod.yml --env-file .env up -d --build
```

Проверить контейнеры:

```bash
docker compose -f docker-compose.prod.yml --env-file .env ps
```

Проверить backend metrics из контейнерной сети:

```bash
docker compose -f docker-compose.prod.yml --env-file .env exec prometheus wget -qO- http://backend:8000/metrics
```

Проверить Prometheus targets через tunnel:

```text
http://localhost:9090/targets
```

Ожидаемые targets:

- `backend`;
- `postgres`;
- `cadvisor`.

В Grafana нужно проверить:

- dashboard `Time Tracking Observability` появился;
- datasource `Prometheus` и `Loki` работают;
- HTTP panels получают данные;
- DB query panels получают данные после запросов к API;
- PostgreSQL panels получают данные;
- container panels получают данные;
- backend и frontend logs отображаются.
